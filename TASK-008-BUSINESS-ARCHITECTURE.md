## 一、业务需求梳理

### 1.1 核心业务场景

#### 场景1: 小区驻点工作全流程

```
触发: 客户经理说"我要去XX小区驻点"
    ↓
准备: AI助手读取小区档案，准备检查清单
    ↓
出发: 文字确认小区名、工作人员姓名
    ↓
检查: 按5项清单逐项采集（拍照+记录）
    ↓
完成: 生成工作简报和巡视记录
```

**5项采集清单**:
1. 小区基本情况 + 大门照片
2. 进网入格（公示栏照片）
3. 供配电方案（手绘供电图）
4. 应急保电方案（发电车接入）
5. 用电检查隐患台账（低配室/竖井）

#### 场景2: 现场照片采集与识别

```
触发: 客户经理发送现场照片
    ↓
保存: 保存到photos/目录
    ↓
识别: AI分析照片内容（设备/隐患）
    ↓
记录: 写入work-record.md
    ↓
反馈: 简短确认信息
```

**照片类型**:
- 配电房: 识别设备类型、环境状态、安全隐患
- 变压器: 型号、外观、锈蚀漏油
- 通用: 主要对象、整体状态

#### 场景3: 文档自动生成

```
触发: "完成采集" / "生成简报"
    ↓
汇总: 收集所有采集数据
    ↓
生成: 创建供电简报（Markdown/Word）
    ↓
推送: 发送文档下载链接
```

**生成文档**:
- `work-record.md` - 驻点工作记录（实时更新）
- `{date}_power-briefing.md` - 供电服务简报

#### 场景4: 信息检索查询

```
触发: "查询阳光小区的变压器信息"
    ↓
检索: 搜索小区README.md和历史记录
    ↓
返回: 结构化信息
```

**查询类型**:
- 小区基本信息
- 设备台账
- 历史问题
- 敏感客户清单

### 1.3 业务数据

#### 数据实体

**小区（Community）**:
- 基本信息: 名称、地址、总户数、楼栋数
- 设备信息: 配电房数量、变压器数量
- 联系人: 物业名称、物业电话
- 敏感客户: 独居老人、孕妇、重病患者清单
- 应急信息: 发电车接入路径、停放位置

**驻点工作记录（WorkRecord）**:
- 时间: 日期、工作人员
- 内容: 5项采集清单完成情况
- 照片: 现场照片集合
- 问题: 发现的隐患和整改状态

**设备台账（Equipment）**:
- 变压器: 型号、容量、状态
- 低配室: 位置、设备清单
- 竖井: 电缆、表箱情况

#### 数据关系

```
小区 1──* 驻点工作记录
小区 1──* 设备台账
小区 1──* 敏感客户
```

### 1.4 业务流程

#### 主流程: 驻点工作

```
开始 → 准备 → 出发 → 检查(循环) → 完成 → 生成文档 → 结束

详细步骤:
1. 开始: 用户触发"去XX小区驻点"
2. 准备: AI读取小区档案，检查历史问题
3. 出发: 文字确认小区名、工作人员姓名
4. 检查: 按5项清单逐项完成（可循环多次）
5. 完成: 确认所有项已采集
6. 生成: 自动/手动触发文档生成
7. 结束: 更新小区档案，标记完成
```

#### 子流程: 照片处理

```
接收照片 → 保存文件 → AI分析 → 记录结果 → 反馈用户

关键点:
- 保存路径: field-data/communities/{小区名}/{YYYY-MM}/photos/
- 命名规范: IMG_{YYYYMMDD}_{HHMMSS}_{SEQ}.jpg
- AI分析: 识别设备、检测隐患、评估状态
```

#### 子流程: 小区切换

```
当前小区A → 保存工作 → 清空上下文 → 加载小区B → 开始新工作

关键点:
- Session变量: current_community
- 数据隔离: 防止A小区数据混入B小区
- 上下文清理: 切换时重置状态
```

### 1.5 业务规则

#### 字段分级（P0/P1/P2）

**P0级（必填，强提醒）**:
- 小区名称（文字确认，避免同音字）
- 工作人员姓名（文字确认）
- 小区地址
- 楼栋数、户数
- 配电房数量、变压器数量
- 公示栏照片（必须包含客户经理姓名和电话）
- 低配室检查记录

**P1级（重要，中等提醒）**:
- 物业公司、物业联系方式
- 敏感客户清单
- 竖井检查记录
- 发电车接入方案

**P2级（选填，轻提醒）**:
- 商业户数
- 特殊人群清单（如无则跳过）
- 充电桩检查（如无则跳过）

#### 隐私保护规则

- 不主动暴露用户身份信息（姓名、电话）
- 工作档案概览要匿名化
- 待办事项不关联具体人员

#### 文件存储规范

```
field-data/
├── communities-index.md        # 小区总索引
├── search-index.md             # 检索索引
├── templates/                  # 文档模板
└── communities/
    └── {小区名}/
        ├── README.md           # 小区档案
        └── {YYYY-MM}/          # 月份目录（强制）
            ├── work-record.md  # 工作记录
            ├── photos/         # 照片
            └── documents/      # 生成文档
```

**强制规范**:
- 必须使用 `YYYY-MM` 月份目录
- 照片必须存放在 `YYYY-MM/photos/`
- 禁止创建日期子目录 `YYYY-MM-DD/`

### 2.3 数据流

#### 主数据流: 驻点工作

```
用户输入（企业微信）
    ↓
OpenClaw Channel 接收
    ↓
Session Manager 更新状态
    ↓
Skills Router 路由到对应Skill
    ↓
Skill 执行业务逻辑
    ↓
调用 Tools（文件操作/AI分析）
    ↓
写入数据层（field-data/）
    ↓
返回响应给用户
```

#### 照片处理数据流

```
用户发送照片
    ↓
photo-collection Skill 接收
    ↓
保存到 field-data/communities/{小区}/{月份}/photos/
    ↓
调用 Kimi K2.5 分析照片
    ↓
分析结果写入 work-record.md
    ↓
返回简短反馈给用户
```

**关键节点**:
- 路径生成（需修复01号Bug）
- 文件名处理（需修复07号Bug）
- 文件写入（需修复09号Bug）
